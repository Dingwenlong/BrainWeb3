# 脑域链医下一阶段执行清单

更新日期：`2026-04-20`

## 1. 本轮迭代定义
本轮迭代建议命名为：

`阶段 A：真实数据闭环`

目标不是铺满全部蓝图，而是在现有样板工程上，完成一条能稳定演示、能持续开发、能支撑后续联邦学习的真实业务主链路：

`上传 EEG -> 落库存档 -> 真实 brain-activity 生成 -> 授权后查看 -> 审计留痕 -> 前端 3D 展示`

## 2. 范围边界
### 本轮必须完成
- 数据资产持久化
- 真实 `brain-activity` 聚合
- 最小授权申请/审批/撤销闭环
- 审计日志落库与查询
- 前端详情页 3D 热力图

### 本轮明确不做
- 完整 DID/VC 体系
- 完整 FATE 联邦训练编排
- 销毁闭环
- 监管端独立门户
- 生产级安全加固

## 4. 推荐拆分方式
建议分成 `4 个 epic`，按顺序推进。

### Epic 1：数据资产持久化
目标：把数据资产从演示态切到真实后台状态管理。

#### 4.1.1 后端改造
- 在 `apps/backend/pom.xml` 增加：
  - `spring-boot-starter-data-jpa`
  - `flyway-core`
  - `mysql-connector-j`
- 新建目录：
  - `apps/backend/src/main/java/com/brainweb3/backend/dataset/persistence`
  - `apps/backend/src/main/resources/db/migration`
- 把 `DatasetRecord` 从临时 record 演进为：
  - `DatasetEntity`
  - `DataAssetProofEntity`
  - `UploadAuditEntity`
- 增加 `DatasetRepository`、`UploadAuditRepository`
- 重写 `DatasetCatalogService`：
  - 去掉内存 `Map`
  - 上传时写数据库
  - 列表/详情从数据库查询
  - 保留现有 `DatasetController` 协议，先不破前端

#### 4.1.2 数据库表建议
- `datasets`
- `data_asset_proofs`
- `upload_audits`

建议最少字段：
- `datasets`: `id`、`subject_code`、`title`、`description`、`owner_organization`、`format`、`original_filename`、`file_size_bytes`、`sampling_rate`、`channel_count`、`sample_count`、`duration_seconds`、`upload_status`、`proof_status`、`training_readiness`、`created_at`、`updated_at`
- `data_asset_proofs`: `dataset_id`、`chain_provider`、`chain_group`、`contract_name`、`contract_address`、`sm3_hash`、`ipfs_cid`、`off_chain_reference`、`chain_tx_hash`、`did_holder`、`access_policy`、`audit_state`
- `upload_audits`: `id`、`dataset_id`、`action`、`status`、`message`、`trace_id`、`created_at`

#### 4.1.3 验收
- 上传后重启应用，数据仍可查询
- 数据列表顺序和详情内容与当前接口兼容
- 上传失败会留下失败状态，而不是直接无记录

### Epic 2：真实 brain-activity 能力
目标：把详情页看到的脑区活跃度从 mock 变成真实计算结果。

#### 4.2.1 EEG 服务改造
- 在 `services/eeg-service` 引入 `mne` 及必要数值依赖
- 将当前单文件 `app.py` 拆成：
  - `services/eeg-service/app.py`
  - `services/eeg-service/brain_activity.py`
  - `services/eeg-service/channel_mapping.py`
  - `services/eeg-service/models.py`
- 固定 8 脑区映射：
  - `LEFT_FRONTAL`
  - `RIGHT_FRONTAL`
  - `LEFT_PARIETAL`
  - `RIGHT_PARIETAL`
  - `LEFT_TEMPORAL`
  - `RIGHT_TEMPORAL`
  - `OCCIPITAL`
  - `CENTRAL_MOTOR`
- 对每个时间窗计算 band power，并输出当前前端已能消费的结构：
  - `regions`
  - `frames`
  - `qualityFlags`

#### 4.2.2 后端代理改造
- 在 `BrainActivityGateway` 和 `DatasetController` 中补齐参数透传：
  - `windowSize`
  - `stepSize`
  - 可选的 `timeStart`
  - 可选的 `timeEnd`
- 在后端加参数校验：
  - `band` 必须是固定枚举
  - `windowSize > 0`
  - `stepSize > 0`
  - 限制单次最大时间范围

#### 4.2.3 建议先不做的高级能力
- 源定位
- 真解剖脑区级医学解释
- 多数据集并排对比
- 在线增量流式回放

#### 4.2.4 验收
- 同一真实文件请求两次，返回结构稳定
- `qualityFlags` 不再固定包含 `mock-data`
- 详情页现有播放、切频段逻辑无需大改即可工作

### Epic 3：最小授权与审计闭环
目标：让敏感数据读取开始受控，避免“所有人都能看见脑区活跃度”。

#### 4.3.1 后端新增模块
新建建议目录：
- `apps/backend/src/main/java/com/brainweb3/backend/access`
- `apps/backend/src/main/java/com/brainweb3/backend/audit`

建议第一版领域对象：
- `AccessRequest`
- `AccessDecision`
- `AuditEvent`
- `UserContext`

#### 4.3.2 第一版接口
建议在 `packages/shared-api/openapi.yaml` 增加：
- `POST /access-requests`
- `GET /access-requests`
- `POST /access-requests/{id}/approve`
- `POST /access-requests/{id}/reject`
- `POST /access-requests/{id}/revoke`
- `GET /audits`

第一版请求头可先简化为：
- `X-Actor-Id`
- `X-Actor-Role`
- `X-Actor-Org`

先不要引入完整登录页和 JWT 登录中心，避免本轮被认证体系拖住。

#### 4.3.3 访问控制策略
- `dataset detail` 可允许登录后查看基础摘要
- `brain-activity` 需要审批通过后才允许读取
- 上传人和资产归属机构默认拥有查看权限
- 审批、撤销、访问失败都写审计

#### 4.3.4 与链的关系
- 第一版可以先以后端数据库为授权真源
- `AccessControl.sol` 保持同步推进基础测试，但不要求本轮把全流程都压到链上
- 如时间允许，再补“审批通过后同步写链”的增强版

#### 4.3.5 验收
- 无权限读取 `brain-activity` 返回明确拒绝
- 审批通过后同一用户可成功读取
- `/audits` 能按 `datasetId` 和 `actorId` 查询关键事件

### Epic 4：前端 3D 热力图与授权态体验
目标：把现有详情页升级成真正可演示的业务页面。

#### 4.4.1 前端页面与组件
建议新增：
- `apps/frontend/src/components/Brain3DHeatmap.vue`
- `apps/frontend/src/pages/AccessRequestsPage.vue`
- `apps/frontend/src/components/AccessStatusBadge.vue`
- `apps/frontend/src/components/AuditTimelineCard.vue`

建议调整：
- `BrainActivityViewer.vue` 变成容器组件或被 `Brain3DHeatmap.vue` 替代
- `DatasetDetailPage.vue` 增加授权态、空态、错误态和质量标记展示
- `router/index.ts` 新增审批页路由
- `types/api.ts` 增加 `AccessRequest`、`AuditEvent` 等类型
- `api/client.ts` 增加访问申请、审批、审计查询接口

#### 4.4.2 交互要求
- 时间轴拖动时 3D 区域着色同步变化
- 频段切换时颜色和数值一起变化
- 悬停脑区显示：
  - 脑区名
  - 当前强度
  - 关联电极
  - 数据质量提示
- 未授权时显示“需审批后查看”，而不是直接请求失败白屏

#### 4.4.3 技术策略
- 如果真脑模型资源还不稳定，本轮允许先做“简化 3D 分区模型”
- 重点是：
  - 有空间感
  - 有 8 分区
  - 有时序着色
  - 能响应 hover 和 playback

#### 4.4.4 验收
- 详情页能稳定展示真实返回数据
- 未授权、有权限、空数据三种状态都有清晰反馈
- 新增审批页可完成最小申请流转

## 5. 推荐接口草案
### 5.1 Access Request
`POST /api/v1/access-requests`

```json
{
  "datasetId": "ds-301",
  "purpose": "research-analysis",
  "requestedDurationHours": 24,
  "reason": "Need to inspect alpha/beta activity before training preview."
}
```

### 5.2 Approve
`POST /api/v1/access-requests/{id}/approve`

```json
{
  "approvedDurationHours": 24,
  "policy": "research-only"
}
```

### 5.3 Audit Query
`GET /api/v1/audits?datasetId=ds-301&actorId=user-001`

### 5.4 Brain Activity 增强参数
`GET /api/v1/datasets/{id}/brain-activity?band=alpha&windowSize=2&stepSize=0.5&timeStart=0&timeEnd=120`

## 7. 开发顺序上的硬依赖
- 没有 `datasets` 持久化，就不适合做真正的权限闭环
- 没有真实 `brain-activity`，3D 热力图只能继续演示化
- 没有最小授权闭环，联邦训练入口没有业务门槛
- 没有审计事件，后续监管与安全验收都无从展开

## 8. 本轮结束后应该交付什么
- 一个能反复上传和查询的真实数据目录
- 一个由真实 EEG 生成的脑区活跃度接口
- 一个最小但可信的审批与授权读取流程
- 一个可演示的 3D 脑区热力图详情页
- 一组能覆盖主链路的自动化测试

## 9. 建议的实际开工点
如果现在就开始开发，我建议第一刀先落在这 3 件事上：

1. 给 `apps/backend` 补数据库依赖和 Flyway。
2. 把 `DatasetCatalogService` 从内存 `Map` 改成 repository 驱动。
3. 在 `services/eeg-service` 里把 mock `build_frames` 替换成真实解析管线的第一版。

这三步一旦完成，后面的授权、审计和前端 3D 联调都会顺很多。
