# 脑域链医下一阶段开发梳理

更新日期：`2026-04-20`

## 1. 当前仓库已验证的基线
- 前端门户可构建，已具备总览页、上传入口、数据详情页、脑区活跃度播放与频段切换。
- Java 后端可通过测试，已提供 `system`、`datasets`、`brain-activity` 三类主路径，并已串起上传、存储、链上存证样板流程。
- Python `eeg-service` 与 `federated-service` 都能启动并通过基础测试，但当前分别停留在 `brain-activity mock` 和 `training preview` 阶段。
- Solidity 合约工程可编译测试，但目前只有 `DataNotary` 进入实际链路，其余合约仍处于独立骨架状态。
- 本地验证已通过：
  - `npm run build:frontend`
  - `mvn -pl apps/backend test`
  - `python -m pytest services/eeg-service/tests services/federated-service/tests`
  - `npm run test:contracts`

## 2. 下一阶段不要分散火力
下一阶段不建议同时全面推进 `DID/VC`、`FATE`、`销毁闭环`、`监管端` 和 `3D 热力图终版`。当前代码已经证明“样板闭环可跑”，但距离“真实业务闭环可演示”还缺几块硬地基：

1. 数据资产仍是内存目录，不是真实数据库记录。
2. `brain-activity` 仍是样例数据，无法支撑真实 EEG 可视化、训练前校验和审计。
3. 授权、审计、访问申请尚未进入真实接口和真实链路。
4. 联邦学习还没有前置授权与数据治理支撑，直接做 FATE 会把复杂度提前引爆。

结论：下一阶段的主题应该是：

`从“上传存证样板闭环”推进到“数据可落库、活跃度可真实生成、访问可受控、过程可审计”的最小业务闭环。`

## 3. 建议的阶段目标
建议把下一阶段定义为一个 `2-3 周` 的聚焦迭代，目标不是铺满全部模块，而是拿下 4 个结果：

1. 上传后的数据资产进入真实数据库，而不是只存在内存。
2. `brain-activity` 从 mock 切换为真实 EEG 聚合结果。
3. 访问申请、审批、撤销、审计事件至少形成最小闭环。
4. 前端详情页从“占位神经舱”升级为可接真实数据的 3D 热力图版本。

做到这 4 点后，后续再接 `FATE`、`DID/VC` 完整版、销毁闭环和监管视图，项目节奏会顺很多。

## 4. 推荐优先级
### P0：把数据主链路从内存态升级为持久化
目标：让上传、查询、存证摘要、状态流转都有真实落点。

建议任务：
- 在 `apps/backend` 引入 `Dataset`、`DataAssetProof`、`StorageReceipt` 的持久化模型。
- 用 MySQL 落地数据资产表、存证表、上传审计表，替换当前内存 `Map` 目录。
- 让上传接口写入数据库，再由列表页和详情页从数据库读取。
- 补齐上传失败、存储失败、链写入失败的状态字段与补偿位。

完成标志：
- 重启后数据不丢失。
- 列表页与详情页读取真实数据库数据。
- 上传记录能够区分 `uploaded / stored / notarized / failed` 等状态。

### P0：把 EEG 服务从 mock 推进到真实聚合
目标：让 `brain-activity` 成为真实能力，而不是演示接口。

建议任务：
- 在 `services/eeg-service` 接入 `MNE` 读取 `EDF/GDF/BDF`。
- 固定一版 `通道 -> 脑区` 映射表，先覆盖当前前端使用的 8 脑区。
- 计算 `delta/theta/alpha/beta/gamma` 的 band power，并按时间窗输出区域强度序列。
- 保留 `qualityFlags`，加入坏数据、缺导联、低采样率等标记。
- 后端代理层补齐 `windowSize`、`stepSize` 参数透传，并约束参数范围。

完成标志：
- 用真实 EEG 文件请求 `/datasets/{id}/brain-activity` 时，返回不再带 `mock-data` 标记。
- 相同输入文件多次请求，输出结构稳定、时间窗数量可预测。
- 前端现有时间轴与频段切换无需改协议即可继续工作。

### P1：补齐最小权限与审计闭环
目标：不要一次做完整零信任体系，先完成“有申请、有审批、有撤销、有审计”。

建议任务：
- 后端增加 `access-requests` 相关接口：申请、审批、拒绝、撤销、查询。
- 先用平台内账户和角色枚举做第一版 RBAC，不急着同步做完整 VC。
- 接入 `AccessControl` 合约或至少先完成后端授权领域模型和审计事件模型。
- 所有上传、查看详情、查看活跃度、审批动作统一产出审计日志。
- 前端补齐最小审批页和访问状态提示。

完成标志：
- 未获批用户不能读取 `brain-activity`。
- 一次申请到审批完成后，前端可看到授权状态变化。
- 审计查询至少能按 `datasetId` 和 `actor` 回看关键动作。

### P1：把前端可视化升级到真实 3D 版本
目标：让详情页的“脑活动可视化”从漂亮占位，升级到有业务价值的交互组件。

建议任务：
- 新增 `Brain3DHeatmap`，用 `Three.js` 或简化分区模型完成 8 脑区着色。
- 保持当前 `频段切换 + 时间轴拖动 + 播放暂停` 交互不变。
- 加入脑区悬停浮层，展示脑区名、当前强度、关联电极。
- 增加空数据、无权限、低质量数据三类兜底态。

完成标志：
- 在详情页可看到真实 3D 或简化 3D 分区热力图。
- 拖动时间轴时着色和右侧指标同步变化。
- 无数据或权限不足时不出现空白黑屏。

## 5. 当前阶段不建议抢跑的内容
这些事情重要，但不建议放进“下一阶段”的主目标里：

- 完整 `DID/VC` 签发、验签、吊销。
- 完整 `FATE` 联邦训练编排和贡献度上链。
- 数据销毁闭环。
- 监管端独立门户。
- 全量安全加固和生产级运维体系。

原因不是它们不重要，而是它们都依赖前面的 4 个地基。当前如果提前推进，文档会更多，真实闭环反而更慢。

## 6. 推荐执行顺序
### 第 1 段：后端数据落库
- 建库与实体建模。
- 替换内存目录。
- 打通上传后的持久化和状态流转。

### 第 2 段：真实 EEG 聚合
- 接入 MNE。
- 固定 8 脑区映射。
- 打通真实 `brain-activity` 接口。

### 第 3 段：权限与审计最小闭环
- 建立访问申请与审批模型。
- 接入授权校验和审计日志。
- 把活跃度读取纳入权限控制。

### 第 4 段：前端体验补齐
- 上 3D 热力图。
- 上审批页和授权态提示。
- 补空态、错态和权限态。

## 7. 需要同步补上的测试
- 后端集成测试：上传成功、上传失败、链写入失败、授权前后读取差异。
- EEG 服务测试：真实样例文件解析、频段输出稳定性、异常文件处理。
- 前端测试：频段切换、时间轴驱动、权限不足和空数据兜底。
- 合约测试：`AccessControl` 和 `DidRegistry` 的基本事件与读取逻辑。

## 8. 下一阶段的退出标准
满足以下条件，就可以认为“下一阶段”完成：

1. 可以上传一个真实 EEG 文件并持久化到数据库。
2. 可以查询该数据集的真实 `brain-activity` 结果。
3. 未授权用户无法读取活跃度，授权后可以读取。
4. 前端详情页能用 3D 热力图展示真实时序变化。
5. 关键主链路具备自动化测试，不再只靠手工演示。

## 9. 一句话判断
如果只能选一个开发主题来代表下一阶段，那应该是：

`先把“真实数据 + 真实权限 + 真实可视化”做成最小闭环，再上联邦学习和完整身份体系。`
