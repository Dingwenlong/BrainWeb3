## 2. 实施策略
- 架构上采用 `monorepo + 模块化单体 + 双 Python 服务 + 合约层`。在 8 周窗口内，不建议从零拆成多套微服务，以降低联调成本。
- Java 侧承接统一业务编排、认证鉴权、链/IPFS 调度、审计汇总与外部网关能力。
- Python 侧拆为两个独立服务：`eeg-service` 负责 EEG 解析/预处理/脑区活跃度计算，`federated-service` 负责 FATE 编排与训练归档。
- 前端维持单一 Vue 3 门户，按角色切换视图，不拆分多套站点。
- 首版坚持“链上只存证明、不存原始明文；前端只读聚合结果、不读原始 EEG”的边界。

## 10. 主要风险与应对
- `FISCO BCOS + FATE` 联调复杂度高：先分别做最小闭环，再进行平台级集成。
- `DID/VC` 实现范围容易膨胀：首版只做注册、签发、校验、吊销四项必需能力。
- `3D 大脑模型` 资源不可控：预留简化分区模型方案，不阻塞交互与数据联调。
- `跨语言链路` 排错成本高：统一请求追踪 ID、审计事件 ID、训练任务 ID。
- `空仓库起步` 容易拖慢第 1 周：必须把工程基线和样板链路作为第一优先级。
