# 脑域链医下一阶段执行清单

更新日期：`2026-04-21`

## 1. 当前判断
当前仓库已经不再是“样板骨架”阶段，而是已经具备以下可运行能力：

- 已完成 `EDF/GDF/BDF 上传 -> 元数据提取 -> 链下存储 -> 链上存证 -> 数据详情查看`
- 已完成 `brain-activity` 真实 EEG 聚合、时间区间查询与 fallback 兜底
- 已完成访问申请、审批、拒绝、撤销、读取门禁与审计事件写入/查询样板闭环
- 已完成数据详情页 `Three.js` 3D 热力图、时间轴、频段切换、脑区悬停与指标联动
- 已完成一键启停脚本：`start-all.ps1 / stop-all.ps1`

下一阶段不应该再继续围绕“把 mock 变成真的”来规划，而应该围绕：

`把当前可演示样板闭环，推进成可持续开发、可正式验收的最小产品闭环。`

## 2. 本轮推荐主题
建议把下一阶段命名为：

`阶段 B：业务闭环收口`

核心目标是补齐 5 个缺口：

1. 正式鉴权，而不是仅靠请求头切换身份
2. 数据资产真实持久化与失败补偿
3. 审计中心与监管视图最小版
4. 联邦学习最小样板
5. 训练页、销毁页、角色化导航等门户收口

## 3. 范围边界
### 本轮必须完成
- `JWT + 最小 RBAC` 正式鉴权闭环
- 数据资产持久化、幂等、失败补偿
- 独立审计页或监管视图最小版
- `FATE` 本地验证环境与最小训练闭环
- 前端训练页 / 审计页 / 角色导航收口

### 本轮建议不做满
- 完整 DID/VC 全套体系
- 完整链上授权真源切换
- 数据销毁全闭环
- 生产级安全加固
- 完整模型治理平台

## 4. P0 / P1 / P2 Backlog
### P0：正式鉴权闭环
目标：把现在基于 `X-Actor-*` 请求头的样板身份切换，升级成最小正式可交付方案。

建议任务：
- 在 `apps/backend` 增加登录态与 token 校验中间层
- 建立最小账户模型：用户、机构、角色、状态
- 把当前 `ActorContextResolver` 从“信任请求头”切到“解析 JWT + 回填用户上下文”
- 保留当前 `owner / approver / researcher / admin` 角色语义，先不扩展复杂角色矩阵
- 所有 `datasets`、`access-requests`、`audits`、`brain-activity` 接口统一走正式鉴权入口
- 前端增加登录绑定与会话持久化，替换当前纯前端身份切换模式

完成标志：
- 未登录不可访问受保护接口
- 登录后根据角色自动看到对应可用入口
- `brain-activity` 与审批动作不再依赖手工传请求头切角色

相关起点：

- [ActorProfileSwitcher.vue](</D:/Devs/BrainWeb3/apps/frontend/src/components/ActorProfileSwitcher.vue:1>)

### P0：数据资产持久化与失败补偿
目标：把当前能跑的上传链路升级成稳定后台能力。

建议任务：
- 明确 `datasets / data_asset_proofs / upload_audits` 的持久化状态机
- 上传流程按阶段落状态：`uploaded / stored / notarized / failed`
- 增加幂等控制，避免同文件重复提交导致脏数据
- 补齐链下存储失败、链上写入失败、元数据提取失败的补偿或重试入口
- 明确真实链下存储主方案：继续以 `MinIO` 为主，还是补真实 `IPFS`
- 补管理接口或脚本，支持重试失败任务

完成标志：
- 服务重启后数据仍完整
- 上传失败有留痕、有状态、有可重试路径
- 至少一条失败补偿路径可验证

### P0：审计中心最小版
目标：把“有审计记录”升级为“能看、能筛、能回放关键动作”。

建议任务：
- 前端新增独立审计页，而不只是在详情页侧栏看时间线
- 支持按 `datasetId / actorId / action / 时间范围` 查询
- 区分关键动作：上传、审批、拒绝、撤销、读取活跃度、训练启动
- 给监管视角保留单独入口或单独过滤模式
- 统一事件文案与状态枚举，避免前后端显示不一致

完成标志：
- 管理者或归属方可独立查询关键审计事件
- 能按数据集追溯主链路动作
- 前端不再依赖详情页局部时间线承担全部审计能力

### P1：FATE 最小训练闭环
目标：不要一上来做完整联邦平台，先拿下“能启动一次、能看状态、能归档结果”的最小版。

## 5. 推荐执行顺序
### Sprint 1
- 正式鉴权闭环
- 数据资产持久化状态机与失败补偿

### Sprint 2
- 审计中心最小版
- 前端门户收口第一轮

### Sprint 3
- `FATE` 最小训练闭环
- 训练页与训练状态面板

### Sprint 4
- 链能力补齐的第一批增强
- DID/VC 或销毁闭环二选一启动

## 6. 本轮需要补的测试
- 鉴权测试：未登录、已登录、越权访问、审批人权限
- 上传状态机测试：成功、失败、重试、幂等
- 审计查询测试：按条件过滤、关键事件完整性
- 联邦训练测试：创建任务、状态轮询、失败重试
- 前端集成测试：登录态、角色导航、训练入口、审计页筛选

## 7. 退出标准
满足以下条件，可以认为“阶段 B”完成：

1. 用户通过正式登录态访问系统，而不是依赖手工切请求头
2. 数据上传后具备清晰状态流转和失败补偿能力
3. 审计页面可以独立回看关键业务动作
4. 至少一条训练样板链路可以被授权后启动并追踪状态
5. 前端已具备总览、审批、详情、审计、训练的最小门户形态

## 8. 一句话优先级判断
如果当前只能做一件最重要的事，那应该是：

`先补正式鉴权和后台状态机，把现有样板系统变成可持续运行的最小产品，再接训练、DID/VC 和更重的链能力。`
