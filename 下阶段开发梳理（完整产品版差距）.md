# 脑域链医下阶段开发梳理（完整产品版差距）

更新日期：`2026-04-22`

## 1. 当前判断
结合现有代码、测试、脚本和 [最终进度总表](</D:/Devs/BrainWeb3/docs/最终进度总表.md:1>)，项目当前已经不是“核心功能未完成”，而是进入了：

`最小真实产品闭环已完成，但距离完整产品版仍差正式化、运营化和最终交付收口。`

当前和“完整产品版”相比，仍有 5 组真实缺口：

1. 联邦学习仍是最小训练闭环，不是完整 FATE 正式版
2. DID / VC 仍是平台内最小治理，不是对外正式身份体系
3. 链能力仍以最小真实业务链路为主，不是完整事件合约与索引体系
4. 安全与运维仍缺生产级强化
5. 最终演示脚本、验收材料、汇报材料还没有完全封包

## 2. 下阶段不要平均用力
如果下一阶段同时全面铺开 `FATE 正式版`、`DID/VC 正式版`、`链能力正式版`、`安全运维强化` 和 `最终交付包装`，结果很容易是每条线都推进了一半，但没有一条线真正收口。

更合理的判断是：

- 当前最紧迫的，不是“功能还能再加多少”，而是“现有平台能不能进入正式环境、能不能稳定演示、能不能支撑验收”
- `联邦学习正式版`、`DID/VC 正式版`、`链能力正式版` 都是重要增强，但它们的价值，会在正式环境、运维基线和交付包装完成后更容易体现

结论：

`下一阶段建议以“正式化与交付收口”为主线，以“联邦学习正式版”为第一增强方向，DID/VC 与链能力正式版作为后续并行增强主题。`

## 3. 建议的阶段主题
建议把下阶段定义为：

`阶段 C：正式化与交付收口`

目标不是继续证明“系统能跑”，而是拿下 4 个结果：

1. 正式环境联调能真实跑通并留档
2. 安全与运维具备最小生产基线
3. 最终演示脚本、验收材料、汇报材料可以打包交付
4. 在此基础上，启动联邦学习正式版第一轮增强

## 4. 推荐优先级
### P0：正式环境联调与验收收口
目标：把当前的本地可运行平台，推进到“可在目标环境中完成真实联调和验收”的状态。

建议任务：
- 固定正式环境变量模板与配置映射
- 逐项跑通 `MySQL / MinIO / FISCO / EEG 服务` 的真实地址联通
- 用正式环境执行 `check-prod-env`、`smoke:backend`、`report:acceptance`
- 固化演示账号、演示顺序、演示数据集和异常兜底话术
- 产出最终验收包：环境清单、功能截图、验证截图、已知限制说明

完成标志：
- 可以在正式环境完成一次完整业务链路演示
- 有可复用的 smoke 结果和验收报告
- 演示脚本不再依赖开发人员临场解释

### P0：安全与运维基线
目标：把当前“能启动、能演示”的工程，推进到“具备最小运维可信度”的状态。

建议任务：
- 增加统一限流和关键接口防刷策略
- 补齐 TLS，评估并预留 mTLS 接入位
- 明确日志、监控、告警的最小方案
- 补齐数据库、对象存储和关键配置的备份策略
- 增加运维脚本：启动检查、健康检查、备份、恢复、日志导出

完成标志：
- 关键服务有健康检查与告警入口
- 至少一条备份/恢复路径被验证
- 关键接口具备最小限流与传输安全约束

### P1：联邦学习正式版第一轮
目标：把“最小训练闭环”升级到“真实 FATE 训练能力的第一阶段”。

建议任务：
- 落一套真实可验证的 FATE 本地/测试环境
- 明确训练编排流程：创建、调度、执行、完成、失败、重试
- 增加训练任务日志、产物归档、模型版本记录
- 补第一版贡献度或训练摘要记录
- 把训练页从样板入口升级成可持续使用的任务工作台

完成标志：
- 至少一条真实训练任务可从创建跑到完成
- 训练状态和结果可在前端稳定查看
- 训练失败具备明确日志和重试入口

### P1：最终交付包装
目标：把当前零散文档和页面能力，收成一个“能交、能讲、能验”的产品包。

建议任务：
- 固化最终版演示脚本
- 补齐验收材料、汇报材料、功能清单和系统架构摘要
- 统一对外用语：平台名称、模块名称、角色名称、链能力描述
- 形成一套可直接用于答辩/验收/汇报的图文材料

完成标志：
- 交付方可以脱离源码仓库直接理解系统结构和验收路径
- 演示、验收、汇报材料之间不再互相矛盾

### P2：DID / VC 正式版
目标：从平台内身份治理，升级到正式可交换身份体系。

建议任务：
- 真实 VC 签名与验签
- 正式吊销注册表
- 对外交换格式定义
- 与当前账户、机构、审批链路对齐

完成标志：
- VC 不再只是平台内状态对象，而具备正式验证能力

### P2：链能力正式版
目标：从“最小真实链路”升级到“完整业务事件与索引治理”。

建议任务：
- 补完整立业务事件合约体系
- 增加事件监听与回放
- 建立正式索引化查询能力
- 明确哪些事件必须上链，哪些保留在审计层

完成标志：
- 不再只有最小链路证明，而是具备结构化事件治理能力

## 5. 推荐执行顺序
### 第 1 段：正式环境与交付收口
- 正式环境联调
- smoke 和验收报告
- 演示脚本与验收包固化

### 第 2 段：安全与运维基线
- 限流
- TLS / mTLS 预留
- 监控、告警、备份、恢复
- 运维脚本补齐

### 第 3 段：联邦学习正式版第一轮
- FATE 环境
- 训练编排
- 日志与结果归档
- 训练页收口

### 第 4 段：身份与链能力正式版
- DID / VC 正式签发与验签
- 吊销注册表
- 完整事件合约与索引治理

## 6. 当前阶段不建议抢跑的事情
这些方向重要，但不建议在下一阶段一开始就作为主目标：

- 全量链事件合约一次性重做
- 完整 DID / VC 对外交换生态一次性铺开
- 联邦学习贡献度、模型治理、链上治理同时并进
- 大规模安全攻防和全套容灾演练

原因不是它们不重要，而是当前最缺的是：

`把现有平台从“开发完成”推进到“正式可交付”，而不是继续扩大功能面。`

## 7. 下阶段退出标准
满足以下条件，可以认为“下阶段”完成：

1. 正式环境联调至少完成一次可复现跑通
2. 最终演示脚本、验收材料、汇报材料形成固定包
3. 关键服务具备最小安全与运维基线
4. 联邦学习正式版第一轮能力可被真实验证

## 8. 一句话判断
如果下阶段只能选一个主题，那应该是：

`先把正式环境、交付材料和运维基线收口，再把联邦学习推进到正式版第一轮，最后补 DID/VC 与链能力的完整正式化。`
